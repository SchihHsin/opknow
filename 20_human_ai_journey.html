任务分工：谁干哪些活

AI（7 步）把「知识搬进答案」——多轮 web_search 找方案、web_fetch 取官方正文、拼二手、起草 msopgen 命令 / 核函数 / Tiling / aclnn 绑定、解读编译报错。

用户（10 步含分支）立项定目标、本地执行（msopgen 生成工程 / build.sh 编包 / 部署 OPP / 集成 / 跑测试）、做 3 个关键决策、2 次脱离 AI 独立求证。

AI ≈ 41%
用户 ≈ 59%
量化口径：按节点计数（AI 7 / 用户 10），非工时。AI 在检索阶段（P2）几乎独占，用户在环境执行与验收阶段（P3 / P5 / P7）独占；这正是「越往本地落地，人的占比越高」。

用户决策依据：判断前必须先拿到什么

三个决策点（图中 ◆ d1/d2/d3）做判断前各自需要的关键信息：

  • d1 选 soc_version：需先 npu-smi info 查到实际芯片型号（如 Ascend910B1）——AI 给不了，必须线下核验。
  • d2 版本对齐：需确认本机 CANN 版本落在 80RC2alpha001 / 80RC3 / 82RC1alpha003 哪条线——因为 add_custom.json 字段与目录路径在版本间会变。
  • d3 精度是否达标：需 CPU / GPU 参考输出做逐元素比对，AI 无法替用户判定「够不够准」。

问题卡点 + 解决方案

  • 卡点 s3（实测受阻）：官方 how-to 整段压在 devguide/opdevg/ SPA 子树，web_fetch 只回导航 + meta、正文抓不到。
    → 解决：转走二手（s4）+ 让用户开浏览器人眼自查（分支 b1，人能看到 AI 抓不到的渲染后正文）。
  • 卡点 s9（编译报错）：build.sh 编包 / 部署 OPP 阶段常因版本、soc_version、算子注册不匹配报错。
    → 解决：把报错日志回灌给 AI 解读（s9），AI 给出改法，用户改后重试（s10）。
  • 卡点：版本散乱贯穿全程（d2）。
    → 解决：用户外部求证 Gitee samples 仓对齐版本（分支 b2）。

④Agent 底层系统工程 · 跳出业务表层看技术实现

前三节是「业务层看到的旅程」；本节下沉到「P2 检索区段里，Agent 内部到底怎么跑」。

4.1 工具调用策略：为什么是有限次检索，而非更多 / 更少

上图 s2–s4 里 AI 没有把检索工具刷到上限，实测 D 任务只搜了 ~1–2 轮就转入拼装。背后是两层机制叠加：

  • 硬上限 = max_uses：web_search 是 Anthropic Messages API 的服务端工具，注册时带 max_uses 封顶（如 5）；超过会返回 max_uses_exceeded，模型不能无限刷。
  • 软停止 = 模型自主判断「够了」：服务端工具在同一轮内自带 loop——模型决定搜什么 → 平台执行并回灌结果 → 模型判断信息是否足够 → 不够再搜、够了就停并生成终答（stop_reason: end_turn；长耗时会先 pause_turn 再续）。所以「调用 N 次」不是写死的常量，而是 min(max_uses, 模型判断收敛点)。
  • 为什么 D 任务收敛得早：实测前 1–2 条结果（知乎 / ai6s.net / MindSpore 官方 / CSDN）已能拼出步骤骨架，边际信息递减；继续搜更多只会拿到同质 CSDN 互抄（D 的二手本就薄），模型据此判断「再搜性价比低」→ 提前停。检索次数少 ≠ 偷懒，而是边际收益判断。

输入优化：用户怎么写指令能显著抬升输出精准度

优化手段对这条旅程的具体作用
补约束：写明「要可照抄的完整命令 + 锁定版本号」逼 AI 在 s5 给出带真实 soc_version 占位说明的 msopgen 命令，而非泛泛描述，减少 d1 的来回。
限范围：写明芯片型号 / CANN 版本（如 910B1 + 8.0.RC3）直接消解 d2 版本对齐这个卡点——AI 不必在多版本间猜，输出一次到位。
限渠道：写明「优先官方文档，拿不到说明受阻、不要编」触发 AI 在 s3 受阻时诚实标注 SPA 抓不到并转二手，而不是用记忆补全一个看起来对、跑不通的步骤。
给反馈：把 s9 的编译报错原文贴回AI 解读的准确度直接取决于报错原文，而非「我编译失败了」这类模糊描述。

4.2 大模型每一轮内部在做什么

把 P2 检索区段拆成模型的内部动作循环，逐轮对应到上图节点。
轮次内部动作对应图节点
第 1 轮规划：解析「Ascend C 自定义算子 + 接 PyTorch」→ 拆成「找官方 how-to → 找工程脚手架 → 找 PyTorch 绑定方式」三个子目标；决定先发 web_search。s2
（服务端）读结果 + 研判：扫一遍搜索摘要，识别出官方页 + 知乎 + MindSpore + CSDN；判断官方页最权威 → 决定 web_fetch 官方 URL。s2→s3
第 2 轮遇阻分支判断：fetch 回来只有 meta、正文缺失 → 识别为 SPA 受阻 → 不编造，改判「靠二手 + 自带知识拼」→ 决定再读 MindSpore / CSDN。s3（受阻）→s4
第 3 轮结果生成：把二手碎片 + 自带知识合成步骤骨架，按「环境→msopgen→核函数→编包→部署→绑定」排序，给占位 + 标注哪些需用户核验（soc_version / 版本）→ 输出，end_turn。s5
关键内部动作 = 「研判信息是否足够」与「遇阻是否编造」两个判断点。D 任务的特殊性在于：官方受阻 + 二手薄 + 自带知识在 Ascend C 深处也不厚 → 模型在第 2 轮就进入「没有强证据」状态，只能给骨架而非可照抄全程，这是综合分低的技术根因。

4.3 完整工程代码逐段解析 示范实现 · 基于公开 Anthropic Messages API，非 Claude 内部源码

下面是一个忠实于「我平常怎么跑」的最小 agent 循环：接收自然语言 → 模型自主调度工具 → 多轮回灌 → 终答。逐段注释对应上图与 §4.1/§4.2。
import anthropic
client = anthropic.Anthropic()

# ① 工具注册：web_search 是服务端工具，max_uses 即 §4.1 的硬上限
TOOLS = [
    {"type": "web_search_20250305", "name": "web_search", "max_uses": 5},
    {"name": "web_fetch", "description": "抓取一个 URL 的正文",   # 客户端工具：演示通用 agent 循环
     "input_schema": {"type": "object",
         "properties": {"url": {"type": "string"}}, "required": ["url"]}},
]

def run_agent(user_question):
    # A. 接收并解析用户原始自然语言输入 —— 直接进 messages，模型负责理解
    messages = [{"role": "user", "content": user_question}]

    while True:
        # B. 调度：模型自己决定这一轮是搜 / 抓 / 还是直接答（§4.2 的「规划」）
        resp = client.messages.create(
            model="claude-opus-4-8", max_tokens=4096,
            system="优先给可照抄命令、锁定版本号；官方抓不到就说明受阻，绝不编造。",
            tools=TOOLS, messages=messages)
        messages.append({"role": "assistant", "content": resp.content})

        # C. 终止条件：模型不再要工具 → 收敛、返回终答（对应图 s5 的 end_turn）
        if resp.stop_reason != "tool_use":
            return "".join(b.text for b in resp.content if b.type == "text")

        # D. 多轮循环：执行客户端工具并把结果回灌（受阻时如实标注，不编造 → 图 s3）
        results = []
        for b in resp.content:
            if b.type == "tool_use" and b.name == "web_fetch":
                body = fetch_url(b.input["url"])
                results.append({"type": "tool_result", "tool_use_id": b.id,
                    "content": body or "（SPA / robots：正文抽不到）"})
        messages.append({"role": "user", "content": results})
A 段
接收解析用户自然语言：原始问句直接进 messages，不做模板化预处理——理解交给模型本身。
B 段
调度逻辑：每轮 messages.create 让模型自主决定搜 / 抓 / 答。web_search 的多轮在服务端同一次调用内闭环（§4.1 软停止）。
C 段
终止逻辑：stop_reason != "tool_use" 即模型判断信息足够、收敛产出终答——对应图中 s5。
D 段
多轮循环 + 受阻处理：客户端工具结果回灌；抓不到正文时显式写「正文抽不到」而非留空或臆造，这是「绝不编造」铁律在代码层的落点（图 s3 受阻）。
诚实边界：上述为公开 Messages API 的忠实示范，用于解释 agent 循环骨架，不是 Claude 产品的内部源码；服务端 web_search 的内部检索 / 排序实现不可见，这里只能就「对外可观测的调用契约」作解。

⑤用户决策节点专项 · 为什么这三处非人不可

图中三个 ◆ 决策点，逐一拆「介入原因 / 前置条件 / 执行动作」。

d1选 soc_version（芯片型号）

介入原因
AI 不知道用户机器实际插的是哪颗 NPU——这是本机环境事实，不在任何文档 / 检索可得范围内，无法自动判定。
前置条件
能访问目标机器、会用 npu-smi info 读硬件信息；知道 soc_version 会直接决定 msopgen 生成的工程能否在本机编译。
执行动作
本地查到型号（如 Ascend910B1）→ 用 ai_core_Ascend910B1 替换 AI 给的占位 → 回填继续。

d2版本对齐（CANN / 路径 / json 字段）

介入原因
D 任务版本散乱是实测短板：80RC2alpha001 / 80RC3 / 82RC1alpha003 之间，add_custom.json 字段与文档目录路径都会变（连官方 URL 从 devguide/opdevg/ 漂到 opdevg/Ascendcopdevg/）。AI 给的步骤未必匹配本机版本。
前置条件
知道本机装的 CANN 确切版本；理解「同一份步骤换个版本可能跑不通」。
执行动作
核对本机版本 → 必要时脱离 AI、去 Gitee samples 仓（分支 b2）对齐对应版本的样例 → 修正后再用。

d3精度是否达标（验收）

介入原因
「算子结果够不够准」是带业务容忍度的价值判断——多大误差可接受取决于下游模型，AI 无法替用户拍板同意 / 驳回。
前置条件
有 CPU / GPU 参考实现的输出做基准；定好误差阈值（如相对误差 < 1e-3）。
执行动作
逐元素比对 → 达标则交付（s14）；不达标则驳回，回到 s7 让 AI 协助改核函数 / Tiling，重走 P4–P7。
