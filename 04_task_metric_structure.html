技术问答任务 · 度量结构
把"同一问题在不同技术栈跑出来的差异"抽象成 7 个可复用维度。任何"在社区里找答案"的任务（算子开发 / 训练推理 / 部署调优…）都能用这套刻度打分、横向比较。
七个度量维度（评分刻度 ●＝1，○＝0，满分 5）
① 版本敏感度Version Sensitivity高=必须查
答案随版本 / 硬件变化的程度。高 → 不能凭记忆，必须取证并锁版本。
② 一手源可达性Source Reachability高=好
官方文档好不好找（几轮命中）+ 好不好抓（静态页 vs JS 单页应用）。
③ 二手生态厚度Ecosystem Depth高=好
高质量二手内容（GitHub / SO / 博客）的丰富度与可信度。
④ 模型先验强度Prior Strength高=好
训练数据中该主题的覆盖密度 → 我能多大程度凭已有知识答。
⑤ 检索成本Retrieval Cost高=差
达到可靠答案所需的 search / fetch 轮数。越高越费力（此维度反向）。
⑥ 答案可验证性Verifiability高=好
能否锁定版本、给出可复现步骤，而非通用化的模糊步骤。
⑦ 综合置信度Final Confidence结果指标
前六维综合而来的产出可信度——是这套度量最终要预测的结果量。一般：②③④高、⑤低 → ⑦高。

怎么用这套度量判断"类似任务"（判定速记）
🔎
先判 ①。版本敏感度高 → 直接进入"必须检索"模式，不允许凭记忆给命令。
📉
②③④ 同时偏低（官方难达 + 二手薄 + 先验弱）→ 预判 ⑤ 检索成本会高、⑥⑦ 会降。这正是 CANN 的画像。
🧭
②③④ 同时偏高 → 一两轮即可收敛、可凭可靠源快速作答。这是 CUDA 的画像。
⚠️
②（可达性）单独低（官方是 SPA / 付费墙 / 抓取失败）→ 即使生态不薄，也要触发兜底：换源、换提取、或标注"全文未读全"。
🎯
⑥ 低（锁不了版本） → 必须向用户澄清版本/硬件，否则置信度封顶为"中"。

这套度量的价值：它把"我答得顺不顺、信不信得过"从一句感觉，拆成了 7 个可观测、可打分、可横向比的维度。同一套刻度套到不同任务（训练/推理、调优）和不同栈（CUDA/昇腾）上，就能得到一张"从用户场景看昇腾社区 AI 可用性"的画像——哪些任务昇腾社区已经好查好用、哪些还需要补官方内容或改善文档可抓取性，一目了然。
