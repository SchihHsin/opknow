## 3. 度量体系（11 项 + 综合；2026-06-09 重新定义）

**核心被测对象**：当我（Claude）回答某栈某开发任务时，能否把一份**正确、具体、可锁版本、可复现**的答案搬进回复，以及花多大力气。这是「AI 可用性」（站在会检索的 AI 助手视角，不是"栈好不好用"）。

**概念模型**：一条"把知识搬进答案"的价值链——**触达（搜得到→抓得到）→ 内容质量（够细→版本清楚）→ 检索成本（摩擦）→ 产出（锁版本/可复现）**。知识有三个来源渠道（官方一手 / 二手社区 / 模型自带知识），**前两关对每个渠道分别看** → "三源 × 两关"网格。

| 渠道 / 段 | 指标 | 说明 |
|---|---|---|
| 官方·触达 | ① 官方可发现性 | 官方文档排名 / 几轮命中 |
| | ② 官方可抓取性 | 静态可抓 vs SPA/robots 拦（CANN 公共常量级短板，统一记 2；CUDA 5） |
| 官方·质量 | ③ 官方正文详尽度 | 取回正文够不够细（参数表/命令/JSON）。**CANN 因②受阻→记「受阻·无法实测」中性态，不打推断分** |
| | ④ 版本清晰度 | 是否容易选对版本（还是多版本号散乱） |
| 二手·触达 | ⑤ 二手丰富度 | GitHub/SO/博客等数量与可发现度 |
| 二手·质量 | ⑥ 二手可信度 / 一致性 | 权威/够新/彼此对得上 |
| 自带·触达 | ⑦ 模型自带知识 | 训练覆盖密度，能否凭知识答（质量面并入本项） |
| 过程 | ⑧ 检索成本 | 轮数 + fetch（反向：越少越好） |
| 产出（因变量） | ⑨ 版本可锁定性 | 能否定到确切版本 |
| | ⑩ 步骤可复现性 | 命令能否照抄跑通 |

### 3a. ⑪ 综合置信度的量化公式（2026-06-10 新增，取代旧「整体研判」主观评端）

用户指旧的 ⑪ 系主观研判、不够严谨，要求**用一个严谨、量化、可复现的公式重算**。现采用 **三源噪声-OR（noisy-OR）**：

```
综合 = K × 版本因子 × 成本因子
K   = 1 − (1−OFF)(1−SEC)(1−OWN)        ← 三渠道任一扛住即可答
OFF = nm①×nm②×nm③   (官方：发现×抓取×详尽)
SEC = nm⑤×nm⑥        (二手：数量×可信)
OWN = nm⑦            (自带知识)
版本因子 = 0.7 + 0.3·nm④   (跨渠道 ±30%)
成本因子 = 0.9 + 0.1·nm⑧   (±10%)
nm(v)=v/5；受阻渠道记 0（≠给③打 1 分，矩阵仍显示"受阻≠低分"中性态）
档位：≥.80高 / ≥.63中高 / ≥.45中 / ≥.24低 / <.24很低
```

- **为什么噪声-OR 而非加权平均**：知识只要一条渠道扛得住就能答；这正是「D 三渠道同时塌才致命、A/B/C 靠某一条兜住仍中高」的数学表达。
- **⑨⑩ 不进公式**（产出/因变量，进了=重复加权）；④⑧ 是跨渠道影响、作全局乘子。
- **八格复算（2026-06-10 由 `score_metrics.py` 代入①–⑧ 算出）**：A.cuda .94 / A.cann **.75**｜B.cuda **.93** / B.cann **.70**｜C.cuda 1.0 / C.cann .77｜D.cuda .88 / D.cann **.41**。
- **据此下修两格**（旧分偏乐观）：**C.cann 高→中高(.77)**（⑦自带仅3、③官方仅3，撑不到高）；**D.cann 中→低(.41)**（OFF≈0/SEC.24/OWN.40→K仅.54）。其余 6 格档位与旧一致（A.cann/B.cann/B.cuda 的分值因②③⑤⑥⑧ 改实测公式略动，但仍中高/高）。
- **自洽校验**：来源构成前三段「有把握来源」之和 ≈ 公式 K（D.cann 50%≈K.54、A.cann 90%≈K.93），两套独立估计互证。
- **落地位置**：17/index 的 `confCompute()`+`confDeriv()` 实时代入①–⑧ 渲染（⑪格悬浮可见）；srcMixBlock 加了 K 自洽校验注脚；18 §4 ⑪ 段换成本公式 + 八格计算表。**两份材料同源、逐字一致**。

### 3b. ①–⑩ 每项指标的量化公式 + `score_metrics.py`（2026-06-10 新增，回应「别的列也别主观判定」）

用户进一步指出：不止 ⑪，**①–⑩ 之前也可能是主观判定**，要求**每个指标有一个量化计算公式、各对应一个 Python 程序去算、再重新跑一遍之前的分**；并举例「二手来源要有可信度排序、每源评分多少、怎么算」。落地：

- **⑥ 时效性已并入公式（2026-06-10，回应「资料时效性算进分数了吗」）**：旧 ⑥ 标尺口头讲"够新"、公式却没算。现加 `recency_factor(dates)`——取各二手**发表日期中位月龄**（today=2026-06），**只罚不奖**（够新是基线、给加分会把临界格虚高地推过整数档）：中位月龄 ≤36 不罚 / ≤48 −0.25 / >48 −0.5。确切发表日**重新 web_search 实测取得**（拿不到的诚实留 None、不参与中位，绝不杜撰）。**实测结论**：CANN 二手多为 2024–2025 新文（A.cann 2025-05、B.cann 2025-06、C.cann 2024-07/2025-06、D.cann 2024-11/2025-12）→ **时效不构成 CANN 二手短板**；仅 CUDA 侧 A(learnopencv 2023-01)、C(siboehm 2022-12) 两格各 −0.25，**均未翻档（⑥ 仍 4）、⑪ 综合与档位全不变**。即时效性是「合理该有的指标」，但代真实日期后只精化输入、不推翻任何结论。
- **因子 A：⑦ 知识截止 gap（2026-06-10，用户问"除时效外还有别的要算进去的吗"后定 A+B 两项）**：⑦ 原是全 11 项**唯一纯自评、无客观锚**。加 `cutoff_gap_factor(churn)`——按相关工具/API 迭代节奏 `churn`{stable0/moderate−0.25/fast−0.5}**只罚不奖**给客观锚：迭代越快、我训练知识越可能落后于最新版本。**得分=clamp(own + gap,1,5)**。这是「时效」洞见在自带知识上的**对称延伸**（⑥算二手时效、⑦算自带知识时效）。churn 由 task_run_log 版本扩散+各生态发布节奏判定。**实测**：CUDA 成熟链多 stable(Nsight/cuBLAS)、CANN 前沿 D.cann(Ascend C) fast −0.5、其余 moderate −0.25——**均未翻档**（自评本已反映知不知道）。

1. **昇腾社区可用性是"任务相关"的，不是一刀切的低。** 成熟工具/通识类（A 转换、B Profiling、C 库选型）两边差距小；**前沿开发（D 自定义算子）差距被拉满**。

   - **可抓子树**（服务端渲染、正文可抓）：`quickstart/`(A，atc 命令+参数表+npu-smi)、`modeldevpt/ptmigr/`(B，torch_npu.profiler 完整代码+参数表)；`doc_center/`(C-ATB ascendtb_0001) **部分可抓**（概述+接口机制有、算子列表/调用码无）。
   - **SPA 子树**（只回导航+meta）：`devaids/`(A-ATC 参考)、`devguide/devtools/`(B-msprof 原生)、`apiref/aolapi/`(C-AOL 算子表)、`devguide/opdevg/`(D-Ascend C)。

4. **版本清晰度是 CANN 跨任务普遍弱项**（多版本号并存）。
